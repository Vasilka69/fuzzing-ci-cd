## 2. Краткая формула проекта

Проект разрабатывает отечественную микросервисную CI/CD-систему, которая автоматизирует сборку, тестирование, доставку и развертывание программного обеспечения. Отличительная особенность системы - наличие отдельного этапа фаззинг-тестирования, в том числе с перспективным применением LLM для генерации более осмысленных входных данных.

Текущий репозиторий содержит рабочее ядро такого fuzzing-этапа:

- AFL++ выполняет coverage-guided fuzzing, corpus management, scheduling и crash triage.
- Custom mutator подключается к AFL++ и подает target'у структурные входные данные.
- Python worker заранее генерирует кандидаты локально в fake mode или через OpenAI-compatible `chat/completions`.
- LLM не вызывается из hot path AFL++; мутатор получает уже готовые inputs через локальный IPC.
- Новые интересные inputs из AFL++ queue возвращаются worker'у как feedback.

Иными словами, `llm-aflpp/` - это прототип/ядро исполняющего сервиса фаззинг-тестирования, а не вся CI/CD-система целиком.

## 3. Актуальность и мотивация

Из отчета НИР и презентации следует несколько причин, почему система актуальна.

1. Современное ПО разрабатывается в условиях высокой динамики требований, поэтому нужны быстрые и надежные инструменты сборки, тестирования и доставки.
2. CI/CD снижает влияние человеческого фактора, ускоряет выпуск новых версий и помогает раньше обнаруживать дефекты.
3. Для российских компаний важны импортозамещение, технологическая независимость и возможность работы без зависимости от зарубежных CI/CD-платформ.
4. Зарубежные решения вроде Jenkins, Azure DevOps, TeamCity и Bamboo могут быть недоступны, ограничены или не соответствовать требованиям сертификации в регулируемых сферах.
5. Локальная отечественная система упрощает контроль безопасности, хранение данных внутри периметра и соответствие нормативным требованиям.
6. Интеграция фаззинг-тестирования повышает вероятность раннего выявления сбоев, ошибок и потенциальных уязвимостей.
7. Использование LLM в фаззинге перспективно для сложных форматов, протоколов и API, где случайные байтовые мутации часто разрушают структуру входа и не доходят до глубокой логики программы.

## 4. Цель и задачи проекта

Общая цель из материалов НИР: разработать исполняющие сервисы инструмента автоматизации непрерывной интеграции и развертывания.

Задачи общей системы:

- проанализировать существующие CI/CD-решения;
- изучить подходы к фаззинг-тестированию;
- спроектировать микросервисную архитектуру системы;
- реализовать исполняющие сервисы системы;
- встроить этап фаззинг-тестирования в пайплайн CI/CD;
- исследовать применение LLM для генерации входных данных;
- настроить взаимодействие сервисов;
- провести тестирование и оценку разработанного функционала.

Цель текущего fuzzing-ядра:

- показать гибридный подход AFL++ + LLM-assisted structured mutation;
- сохранить производительность AFL++ за счет асинхронного вынесения LLM из hot path;
- дать воспроизводимый fake-режим без сети и real-режим через OpenAI-compatible endpoint;
- обеспечить feedback loop от AFL++ queue обратно к генератору кандидатов;
- подготовить основу, которую можно обернуть в исполняющий микросервис CI/CD.

## 5. Общая архитектура CI/CD-системы

По приложенной схеме система делится на три крупные зоны:

1. Управляющий уровень.
2. Исполняющие сервисы.
3. Внешние системы.

### 5.1. Управляющий уровень

Управляющий уровень включает:

- UI;
- мастер-сервис;
- шину данных.

UI - веб-интерфейс, через который пользователь работает с системой. В материалах НИР указано, что UI планируется/реализуется как React-приложение.

Мастер-сервис - центральный управляющий микросервис и единая точка входа для UI. Он:

- принимает REST-запросы от фронтенда;
- проверяет права доступа;
- управляет пайплайнами, этапами и заданиями;
- хранит или обновляет состояние выполнения;
- не выполняет тяжелые операции самостоятельно;
- делегирует работу исполняющим сервисам через шину данных.

Шина данных - асинхронный слой взаимодействия между мастер-сервисом и исполняющими микросервисами. По отчету НИР и пользовательскому описанию роль шины выполняет Apache Kafka. Мастер-сервис публикует задания в топики Kafka, а исполняющие сервисы подписываются на нужные топики, выполняют работу и возвращают результаты обратно через Kafka.

В отчете НИР встречается вариант схемы, где вместо "шины данных" на управляющем уровне нарисован "балансировщик нагрузки". В актуальном описании пользователя акцент сделан именно на шине данных/Kafka как канале постановки задач исполняющим сервисам.

### 5.2. Исполняющие сервисы

Исполняющие сервисы - stateless машины выполнения заданий. Они не являются источником истины по пайплайнам и не должны хранить долгоживущее состояние выполнения как основную БД. Их задача - получить задание, выполнить его, собрать результат и отправить статус обратно управляющему сервису.

По приложенной схеме выделены следующие исполняющие сервисы:

- сборочный сервис;
- сервис фаззинг-тестирования;
- сервис непрерывного развертывания;
- сервис исполнения скриптов.

В отчете НИР также присутствуют:

- сервис интеграции с системой контроля версий;
- внутреннее хранилище исходного кода.

Эти элементы можно трактовать как расширенный вариант архитектуры: VCS integration service забирает код из внешней системы контроля версий и помещает его во внутреннее хранилище, после чего сборочный сервис берет код оттуда.

### 5.3. Внешние системы

Внешние системы на схеме:

- система контроля версий;
- менеджер репозиториев;
- целевой сервер для развертывания.

Система контроля версий хранит исходный код. Это может быть Git, Mercurial или другой VCS. В старых seed-данных шаблонов заданий есть шаблоны `vsc/git` и `vsc/mercurial`.

Менеджер репозиториев хранит собранные артефакты. С ним взаимодействуют сборочный сервис, сервис фаззинг-тестирования, сервис непрерывного развертывания и сервис исполнения скриптов.

Целевой сервер для развертывания - среда, куда CD-сервис доставляет и разворачивает готовое приложение.

## 6. Взаимодействие сервисов в общей системе

Типовой поток выполнения пайплайна:

1. Пользователь в UI создает или запускает пайплайн.
2. UI отправляет запрос в мастер-сервис по HTTP/REST.
3. Мастер-сервис проверяет права, валидирует параметры и создает запись о запуске.
4. Мастер-сервис публикует задания в Kafka.
5. Нужный исполняющий сервис получает задание из своего топика.
6. Исполняющий сервис обращается к внешним системам или внутренним артефактам, если это необходимо.
7. Исполняющий сервис выполняет работу: сборку, фаззинг, развертывание или пользовательский скрипт.
8. По завершении сервис отправляет результат через Kafka.
9. Мастер-сервис обновляет состояние задания и сохраняет результаты в БД.
10. UI отображает пользователю статус, логи, артефакты, ошибки и результаты.

Преимущества такого взаимодействия:

- мастер-сервис не блокируется на долгих операциях;
- каждый исполняющий сервис можно масштабировать независимо;
- Kafka снижает связность компонентов;
- задачи не теряются при временной недоступности отдельных исполнителей;
- исполняющие сервисы можно обновлять и развивать независимо.

## 9. Модель данных общей CI/CD-системы

В отчете НИР описана ER-диаграмма, а в `Прочее/migrations/` сохранилась более конкретная SQL-схема. Каталог `Прочее/` сейчас не активная часть fuzzing MVP, но миграции полезны для понимания ранней модели данных.

### 9.1. Основные сущности из отчета

`pipeline` - описание конвейеров:

- `id` - уникальный идентификатор;
- `name` - наименование;
- `description` - описание;
- `trigger_id` - триггер запуска.

`stage` - этапы внутри конвейера:

- `id`;
- `pipeline_id` - внешний ключ на pipeline;
- `order` - порядок этапа;
- `name`;
- `description`.

`job` - конкретные работы в рамках этапа:

- `id`;
- `stage_id` - внешний ключ на stage;
- `order` - порядок работы;
- `status` - статус выполнения;
- `script` - скрипт работы;
- `is_script_primary` - флаг основного скрипта.

`job_history` - история выполнения заданий:

- `id`;
- `job_id`;
- `duration`;
- `start_date`;
- `logs`;
- `additional_data` в JSON.

`job_template` - шаблоны заданий:

- `id`;
- `path`;
- `params_template`.

`job_params` - параметры, используемые с шаблонами:

- `id`;
- `job_template_id`;
- `params`.

### 9.2. Дополнительные сущности из SQL-миграций

В старой SQL-схеме также есть:

`folder`:

- `id`;
- `name`;
- `description`;
- `parent_id` - ссылка на родительскую папку.

`trigger`:

- `id`;
- `name`;
- `pipeline_id`.

В SQL-версии `pipeline` связан с `folder_id`, а не напрямую с `trigger_id`. Это означает, что модель в отчете и миграции не полностью совпадают. Для актуальной реализации управляющего сервиса нужно будет выбрать один согласованный вариант.

### 9.3. Шаблоны заданий из старых seed-данных

В `V0002__data.sql` есть шаблоны:

- `vsc/git` с параметрами `url`, `branch`, `login`, `password`;
- `vsc/mercurial` с параметрами `url`, `branch`, `login`, `password`;
- `build/maven` с параметрами `version`, `args`;
- `build/gradle` с параметрами `version`, `args`;
- `build/javac` с параметрами `version`, `args`;
- `build/gcc` с параметрами `version`, `args`;
- `fuzzing` с параметром `function`;
- `deploy/windows/cmd` с SSH-параметрами и `script`;
- `deploy/linux/bash` с SSH-параметрами и `script`.

Для текущего fuzzing-ядра шаблон `fuzzing` пока слишком общий. В будущей версии ему потребуется расширение: target command, seeds, dictionary, time budget, memory limit, sanitizer mode, LLM mode, endpoint settings, artifact/output policy.

## 10. Место fuzzing-сервиса в CI/CD-системе

Сервис фаззинг-тестирования должен быть одним из исполняющих микросервисов. Он получает задачу через Kafka от мастер-сервиса, выполняет фаззинг и возвращает результат.

Текущий код реализует не Kafka-обвязку, а вычислительное ядро такого сервиса:

- сборку target/harness;
- запуск AFL++;
- подключение custom mutator;
- локальный fake/real LLM worker;
- IPC между мутатором и worker'ом;
- feedback loop;
- сохранение AFL++ output и feedback artifacts.

Будущая микросервисная обвязка должна добавить:

- Kafka consumer для получения fuzzing-задач;
- Kafka producer для публикации статусов и результатов;
- адаптер к менеджеру репозиториев;
- подготовку workspace;
- запуск текущего `llm-aflpp` pipeline как дочернего процесса или библиотечного workflow;
- сбор результатов из `output/`, `runtime/`, crash directories и логов;
- очистку временного состояния после завершения задачи.

Такой сервис должен оставаться stateless в смысле бизнес-состояния: информация о пайплайнах, history, пользователях и настройках остается в мастер-сервисе/БД, а fuzzing-сервис хранит только временный workspace и возвращаемые артефакты.

## 11. Что именно реализовано в текущем репозитории

Активный проект находится в `llm-aflpp/`.

Vendored AFL++ dependency находится в `AFLplusplus/`. Ее не нужно менять без отдельной задачи.

Текущее состояние по `TASKS.md`: MVP практически готов. Реализованы сборка, fake pipeline, real-mode worker, IPC `G`/`A`, fallback без worker и feedback persistence. Оставшиеся задачи относятся в основном к метрикам, сравнению с baseline, чистке логирования и экспериментам.

### 11.1. Активные пути

| Путь | Назначение |

## 12. Архитектура текущего fuzzing-ядра

Упрощенная схема:

```text
targets/dsl/prompt.txt
targets/dsl/seeds/
runtime/discovered/
        |
        v
+-----------------------------+
| Python worker               |
| llm_mutator_server.py       |
| fake / real LLM generation  |
| queue of ready candidates   |
+-------------+---------------+
              ^
              | IPC:
              | G = get candidate
              | A = add feedback
              v
+-----------------------------+
| AFL++ custom mutator        |
| afl_llm_mutator.c           |
| afl_llm_mutator.so          |
+-------------+---------------+
              |
              v
+-----------------------------+
| AFL++ afl-fuzz              |
| coverage, corpus, crashes   |
+-------------+---------------+
              |
              v
+-----------------------------+
| DSL target                  |
| target_dsl.c / target_dsl   |
+-----------------------------+
```

Ключевое архитектурное решение: LLM-вызовы вынесены из hot path AFL++. Функция `afl_custom_fuzz()` не делает HTTP-запросы и не ждет модель. Она только быстро пытается взять готовый candidate из локального IPC. Если candidate отсутствует, используется локальная fallback-мутация.

## 13. AFL++ custom mutator

Файл: `llm-aflpp/src/mutator/afl_llm_mutator.c`

Build artifact: `llm-aflpp/build/afl_llm_mutator.so`

### 13.1. Назначение

Custom mutator связывает AFL++ с внешним генератором структурных inputs. Он:

- подключается к worker'у по TCP или Unix socket;
- запрашивает готовые candidates через opcode `G`;
- смешивает LLM/fake candidate с текущим seed;
- отдает результат AFL++;
- при отсутствии worker'а или пустой очереди делает локальную fallback-мутацию;
- отправляет новые queue entries обратно worker'у через opcode `A`.

### 13.2. Реализованные AFL++ callbacks

В коде реализованы:

- `afl_custom_init`;
- `afl_custom_fuzz_count`;
- `afl_custom_splice_optout`;
- `afl_custom_fuzz`;
- `afl_custom_describe`;
- `afl_custom_post_process`;
- `afl_custom_queue_get`;
- `afl_custom_havoc_mutation_probability`;
- `afl_custom_havoc_mutation`;
- `afl_custom_queue_new_entry`;
- `afl_custom_deinit`.

Минимальные callbacks из PRD выполнены, плюс добавлены optional hooks для havoc и queue filtering.

### 13.3. Состояние mutator'а

Структура состояния содержит:

- `sock` - активный socket к worker'у;
- `seed` - seed для локального PRNG;
- `out_buf` и `out_cap` - выходной буфер до 1 MB;
- `endpoint` - адрес worker'а;
- `last_desc` - описание последней мутации для AFL++;
- `requests` - количество вызовов мутации;
- `hits` - количество успешных получений candidate;
- `misses` - количество fallback/miss;
- `last_reconnect_attempt` - ограничение частоты reconnect.

Счетчики `requests`, `hits`, `misses` уже есть в памяти, но пока не экспортируются как полноценная метрика. Это отмечено в `TASKS.md` как P1.

### 13.4. Поддерживаемые endpoints

Адрес берется из:

1. `LLM_MUTATOR_ADDR`;
2. `LLM_MUTATOR_SOCK`;
3. default `tcp://127.0.0.1:15333`.

Поддерживаются:

- TCP loopback вида `tcp://127.0.0.1:15333`;
- Unix socket path;
- abstract Unix socket с префиксом `@`.

Для TCP hostname `localhost` внутри mutator'а заменяется на `127.0.0.1`.

### 13.5. Fallback-мутация

Если worker недоступен, очередь пуста или произошла ошибка IPC, mutator не останавливает AFL++. Он использует дешевую локальную мутацию:

- копирует исходный seed в выходной буфер;
- выполняет 1-4 простые операции;
- может flip'нуть бит;
- может прибавить небольшое значение к байту;
- может вставить printable ASCII символ;
- может скопировать байт из одной позиции в другую.

Для пустого входа fallback возвращает один байт `A`.

### 13.6. Смешивание candidate с seed

Если worker вернул candidate:

- candidate копируется в выходной буфер;
- размер ограничивается `max_size` и внутренним лимитом;
- если текущий seed непустой и candidate длиннее 2 байт, prefix из seed частично накладывается на начало candidate;
- `last_desc` получает вид `llm-cache-hit:size=<N>`.

Такая стратегия пытается сохранить связь с текущим seed AFL++ и одновременно использовать структурный input от worker'а.

### 13.7. Feedback в worker

Когда AFL++ добавляет новый интересный input в queue, вызывается `afl_custom_queue_new_entry`. Mutator:

- читает файл нового queue entry;
- ограничивает размер до 65535 байт;
- отправляет payload worker'у через opcode `A`;
- не блокирует основную логику при невозможности отправки.

Это и есть обратная связь AFL++ -> worker.

## 14. Python worker

Файл: `llm-aflpp/src/worker/llm_mutator_server.py`

### 14.1. Назначение

Worker - отдельный процесс, который заранее генерирует inputs и держит их в очереди. Он снимает с AFL++ необходимость ждать LLM latency.

Worker умеет:

- запускаться в fake mode без внешнего API;
- запускаться в real LLM mode через OpenAI-compatible endpoint;
- слушать TCP/Unix socket;
- обслуживать IPC операции `G` и `A`;
- держать bounded queue кандидатов;
- запускать несколько producer-потоков;
- читать prompt из файла и перечитывать его при изменении;
- загружать seed corpus из стартовой директории и feedback-директории;
- сохранять feedback samples;
- опционально сохранять raw generated candidates.

### 14.2. Основные настройки worker'а

| Переменная | Значение по умолчанию | Назначение |
| --- | --- | --- |
| `LLM_MUTATOR_ADDR` | `tcp://127.0.0.1:15333` | Worker IPC address. |
| `LLM_MUTATOR_SOCK` | не задан | Legacy alias для address. |
| `LLM_MUTATOR_PROMPT_FILE` | `targets/dsl/prompt.txt` | Prompt file. |
| `LLM_MUTATOR_SEED_DIR` | `targets/dsl/seeds` | Стартовые examples. |
| `LLM_MUTATOR_DISCOVERED_DIR` | `runtime/discovered` | Feedback corpus. |
| `LLM_MUTATOR_LOG_CANDIDATES_DIR` | не задан | Optional raw candidate log. |
| `LLM_MUTATOR_QUEUE_SIZE` | `128` | Размер очереди candidates. |
| `LLM_MUTATOR_WORKERS` | `2` | Число producer threads. |
| `LLM_MUTATOR_MAX_SAMPLE_SIZE` | `65535` | Максимальный размер seed/feedback sample. |
| `LLM_MUTATOR_MAX_CANDIDATE_CHARS` | `2048` | Максимальный размер LLM text result. |
| `LLM_API_TIMEOUT` | `20` | Timeout HTTP-запроса к LLM. |
| `LLM_API_URL` | не задан | Включает real LLM mode. |
| `LLM_API_KEY` | не задан | Bearer token, если нужен. |
| `LLM_MODEL` | `gpt-4.1-mini` | Имя модели для real endpoint. |

### 14.3. Fake mode

Fake mode включается, когда `LLM_API_URL` пустой.

В fake mode worker генерирует синтаксически похожие DSL-программы из:

- `MODE FAST/SLOW/DEBUG`;
- `SET A/B/C <int>`;
- `COPY`;
- `XOR`;
- `APPEND`;
- `CHECK`;
- `LOOP`;
- иногда `CRASH NO/MAYBE/NOW`.

Fake mode нужен для воспроизводимости, демонстрации и тестов без API keys, интернета и локальной модели.

### 14.4. Real LLM mode

Real mode включается, когда задан `LLM_API_URL`.

Worker отправляет POST-запрос на OpenAI-compatible `chat/completions` endpoint с JSON:

- `model`;
- `messages`;
- `temperature: 0.9`;
- `max_tokens: 400`.

Если задан `LLM_API_KEY`, он передается как `Authorization: Bearer <key>`.

Ответ читается из `choices[0].message.content`, кодируется в UTF-8, обрезается по `LLM_MUTATOR_MAX_CANDIDATE_CHARS` и `LLM_MUTATOR_MAX_SAMPLE_SIZE`.

### 14.5. Prompt construction

Worker строит сообщения из:

- системной инструкции: вернуть один компактный fuzzing input без Markdown и пояснений;
- содержимого `prompt.txt`;
- до 3 случайных примеров из seed/feedback corpus;
- требования сгенерировать один новый candidate до 2 KB.

Prompt перечитывается при изменении `mtime` файла, поэтому его можно менять без перезапуска worker'а.

### 14.6. Feedback persistence

При получении opcode `A` worker:

- читает длину payload;
- валидирует размер;
- сохраняет sample в `LLM_MUTATOR_DISCOVERED_DIR`;
- добавляет sample в `seed_corpus`;
- логирует `accepted feedback sample`.

Файлы feedback имеют вид:

```text
queue_<timestamp_ms>_<counter>.txt
```

В текущей версии dedupe нет, ограничение количества persisted samples тоже пока не реализовано. Это отмечено в backlog.

## 15. IPC-протокол между mutator и worker

Транспорт:

- TCP loopback по умолчанию: `tcp://127.0.0.1:15333`;
- Unix socket path;
- abstract Unix socket.

Операция `G` - получить готовый candidate:

```text
client -> server: 1 byte opcode 'G'
server -> client: uint32 big-endian length
server -> client: payload, если length > 0
```

`length == 0` означает cache miss / очередь пуста.

Операция `A` - добавить feedback sample:

```text
client -> server: 1 byte opcode 'A'
client -> server: uint32 big-endian length
client -> server: payload
```

Ограничения:

- mutator отправляет feedback до 65535 байт;
- worker отклоняет feedback, если длина `<= 0` или больше `MAX_SAMPLE_SIZE`;
- mutator не делает сетевые LLM-запросы;
- ошибки IPC приводят к drop/reconnect и fallback-мутации, а не к остановке AFL++.

## 16. DSL target

Файл: `llm-aflpp/targets/dsl/target_dsl.c`

Build artifacts:

- `llm-aflpp/build/target_dsl` - AFL-instrumented target;
- `llm-aflpp/build/target_dsl_cc` - host target для smoke-test без AFL++.

### 16.1. Назначение DSL target

DSL target - демонстрационная программа, на которой проверяется идея структурного фаззинга. Она читает stdin, парсит line-based DSL и имеет скрытый crash-path, достижимый только при осмысленной комбинации команд и значений.

Target не является конечной боевой целью. Он нужен как controlled demo target для MVP, чтобы:

- быстро собираться;
- иметь понятный формат входа;
- давать AFL++ coverage;
- иметь проверяемый crash-path;
- позволять сравнивать обычные байтовые мутации и структурные кандидаты.

### 16.2. Runtime model

Target реализует маленькую VM:

- регистры `A`, `B`, `C`;
- режим `FAST`, `SLOW`, `DEBUG`;
- `loop_value`;
- битовую маску `seen`;
- текстовый буфер `text[256]`.

В AFL++ режиме используется `__AFL_LOOP(UINT_MAX)`, то есть target persistent-friendly. В host smoke mode макросы AFL++ заменяются fallback-реализацией, которая выполняет один проход.

### 16.3. Грамматика DSL

Допустимые команды из prompt:

```text
MODE FAST
MODE SLOW
MODE DEBUG
SET A <int>
SET B <int>
SET C <int>
COPY A B
COPY B C
COPY C A
XOR A <int>
XOR B <int>
XOR C <int>
APPEND <word>
CHECK HELLO
CHECK MAGIC
CHECK PLEASE
CHECK FIZZ
CHECK OPEN
CHECK WORLD
LOOP <int>
CRASH NO
CRASH MAYBE
CRASH NOW
```

Фактический parser в C принимает `COPY <dst> <src>` для любых регистров `A/B/C`, а не только три варианта из prompt.

### 16.4. Check-логика

`CHECK` выставляет bits в `seen`:

- `HELLO` при `MODE_FAST`;
- `MAGIC` при `MODE_DEBUG` и `A == 1337`;
- `PLEASE` при `B == A ^ 1364`;
- `FIZZ` если `text` содержит `open`;
- `OPEN` если `text` содержит `open`;
- `WORLD` если `text` содержит `world`.

### 16.5. Скрытый crash-path

`CRASH NOW` вызывает `abort()`, только если выполнены все условия:

- режим `MODE_DEBUG`;
- `loop_value == 7`;
- `A == 1337`;
- `B == 1337 ^ 1364`, то есть `109`;
- `C == 0x4141`, то есть `16705`;
- `text` содержит `open`;
- bits `MAGIC`, `PLEASE`, `FIZZ`, `OPEN` выставлены, то есть `(seen & 0x1e) == 0x1e`.

Контрольный input:

```text
MODE DEBUG
SET A 1337
SET B 109
SET C 16705
APPEND open
CHECK MAGIC
CHECK PLEASE
CHECK FIZZ
CHECK OPEN
LOOP 7
CRASH NOW
```

Smoke-команда:

```bash
cd llm-aflpp
printf 'MODE DEBUG\nSET A 1337\nSET B 109\nSET C 16705\nAPPEND open\nCHECK MAGIC\nCHECK PLEASE\nCHECK FIZZ\nCHECK OPEN\nLOOP 7\nCRASH NOW\n' | ./build/target_dsl_cc
```

Ожидаемый результат - аварийное завершение через `SIGABRT`.

## 17. Prompt, dictionary и стартовые seeds

### 17.1. Prompt

`llm-aflpp/targets/dsl/prompt.txt` описывает DSL-формат и требует:

- возвращать только raw program text;
- использовать одну команду на строку;
- генерировать 3-10 строк;
- сохранять синтаксис валидным;
- варьировать константы и порядок команд;
- предпочитать семантически осмысленные комбинации случайному мусору.

Это важно, потому что LLM должна помогать AFL++ проходить парсер и доходить до логики target'а.

### 17.2. Dictionary

`llm-aflpp/targets/dsl/dsl.dict` содержит токены:

- команды: `MODE`, `SET`, `COPY`, `XOR`, `APPEND`, `CHECK`, `LOOP`, `CRASH`;
- режимы: `FAST`, `SLOW`, `DEBUG`;
- checks: `HELLO`, `MAGIC`, `PLEASE`, `FIZZ`, `OPEN`, `WORLD`;
- crash flags: `NOW`, `NO`, `MAYBE`;
- регистры: `A`, `B`, `C`.

Dictionary используется AFL++ через параметр `-x targets/dsl/dsl.dict`.

### 17.3. Seeds

`seed01.txt`:

```text

```

`seed02.txt`:

```text
MODE DEBUG

CHECK MAGIC

LOOP 3
```

`seed03.txt`:

```text
MODE SLOW
```

Seeds дают AFL++ и worker'у минимальные валидные примеры DSL.

## 18. Сборка и запуск текущего ядра

### 18.1. Сборка AFL++

Если `AFLplusplus/afl-fuzz` и `AFLplusplus/afl-clang-fast` еще не собраны:

```bash
cd AFLplusplus
make source-only
```

Если LLVM не находится автоматически:

```bash
make LLVM_CONFIG=llvm-config-18 source-only
```

### 18.2. Сборка проекта

```bash
cd llm-aflpp
make all
```

Makefile собирает:

- `build/target_dsl` через `../AFLplusplus/afl-clang-fast`;
- `build/afl_llm_mutator.so` через host compiler `cc`.

### 18.3. Smoke-сборка

```bash
cd llm-aflpp
make smoke
```

Собирает:

- `build/target_dsl_cc`;
- `build/afl_llm_mutator.so`.

### 18.4. IPC smoke без AFL++

```bash
cd llm-aflpp
make ipc-smoke
```

Тест:

- запускает worker в fake mode на `tcp://127.0.0.1:15334`;
- делает `G` request;
- ждет непустой candidate;
- отправляет `A` feedback;
- проверяет, что feedback записан во временную `discovered` директорию.

Ожидаемый вывод похож на:

```text
ipc smoke ok: received ... bytes; persisted feedback in /tmp/llm-aflpp-ipc-smoke-.../discovered
```

### 18.5. Fake pipeline

```bash
cd llm-aflpp
AFL_I_DONT_CARE_ABOUT_MISSING_CRASHES=1 AFL_SKIP_CPUFREQ=1 AFL_NO_UI=1 timeout 8s ./scripts/run_fake.sh
```

`run_fake.sh`:

- вызывает `make all`;
- создает `runtime` и `output`;
- выставляет `LLM_MUTATOR_ADDR`;
- задает prompt, seed dir и discovered dir;
- запускает Python worker;
- запускает `afl-fuzz` с `AFL_CUSTOM_MUTATOR_LIBRARY`;
- включает `AFL_CUSTOM_MUTATOR_ONLY=1` по умолчанию;
- передает словарь `dsl.dict`.

Default output:

- AFL++ output: `llm-aflpp/output/fake`;
- feedback: `llm-aflpp/runtime/discovered`.

### 18.6. Real LLM pipeline

```bash
cd llm-aflpp
export LLM_API_URL="http://127.0.0.1:11434/v1/chat/completions"
export LLM_MODEL="qwen3:8b"
unset LLM_API_KEY
export NO_PROXY=127.0.0.1,localhost
export no_proxy=127.0.0.1,localhost
AFL_I_DONT_CARE_ABOUT_MISSING_CRASHES=1 AFL_SKIP_CPUFREQ=1 AFL_NO_UI=1 timeout 30s ./scripts/run_real_llm.sh
```

`run_real_llm.sh` требует, чтобы `LLM_API_URL` был задан. Остальная логика аналогична fake run, но output по умолчанию идет в `llm-aflpp/output/real`.

### 18.7. Fallback без worker

Проверка, что AFL++ не ломается при недоступном worker:

```bash
cd llm-aflpp
AFL_I_DONT_CARE_ABOUT_MISSING_CRASHES=1 AFL_SKIP_CPUFREQ=1 AFL_NO_UI=1 \
AFL_CUSTOM_MUTATOR_LIBRARY="$PWD/build/afl_llm_mutator.so" \
AFL_CUSTOM_MUTATOR_ONLY=1 \
timeout 4s ../AFLplusplus/afl-fuzz -i targets/dsl/seeds -o output/no_worker -x targets/dsl/dsl.dict -- build/target_dsl
```

В этом режиме mutator будет пытаться подключиться к worker'у, но при miss/ошибке использовать локальную fallback-мутацию.

| `LLM_API_KEY` | Bearer token для endpoint. | Не задан |
| `LLM_MODEL` | Model name для real mode. | `gpt-4.1-mini` |
| `LLM_API_TIMEOUT` | Timeout LLM HTTP request в секундах. | `20` |
| `PYTHON` | Python interpreter для scripts и Makefile. | `python3` |
| `NO_PROXY`, `no_proxy` | Proxy bypass для localhost endpoints. | Рекомендуется `127.0.0.1,localhost` для локальных LLM |

## 20. Как fuzzing-ядро должно встраиваться в исполняющий микросервис

Текущий `llm-aflpp` можно рассматривать как внутренний runner. Для полноценного микросервиса нужна обвязка вокруг него.

### 20.1. Входная Kafka-задача

Потенциальный состав задания на фаззинг:

```json
{
  "job_id": "uuid",
  "pipeline_id": "uuid",
  "stage_id": "uuid",
  "project": {
    "repository_url": "https://...",
    "commit": "abc123",
    "branch": "main"
  },
  "artifact": {
    "repository": "repo-manager",
    "path": "group/app/build.tar.gz"
  },
  "fuzzing": {
    "target_command": "./target @@",
    "input_mode": "stdin",
    "seeds_path": "seeds/",
    "dictionary_path": "dicts/input.dict",
    "timeout_seconds": 300,
    "memory_limit_mb": 1024,
    "use_llm_mutator": true,
    "llm_mode": "fake-or-real",
    "llm_model": "model-name"
  }
}
```

Это не реализованный контракт, а рекомендуемая структура на основе текущих потребностей runner'а.

### 20.2. Жизненный цикл задания fuzzing-сервиса

1. Получить сообщение из Kafka.
2. Создать временный workspace.
3. Получить исходный код или артефакт из VCS/внутреннего хранилища/менеджера репозиториев.
4. Подготовить target/harness.
5. Подготовить seeds, dictionary, prompt.
6. Выбрать режим: baseline AFL++, fake LLM-mutator или real LLM-mutator.
7. Запустить AFL++ на заданный бюджет времени/ресурсов.
8. Собирать логи и промежуточные статусы.
9. По завершении разобрать `fuzzer_stats`, crashes, queue, discovered feedback, generated candidates.
10. Сформировать summary.
11. Сохранить артефакты в менеджер репозиториев или другое хранилище.
12. Отправить результат в Kafka для мастер-сервиса.
13. Очистить временный workspace.

### 20.3. Выходной результат fuzzing-сервиса

Потенциальный состав результата:

```json
{
  "job_id": "uuid",
  "status": "success-or-failed",
  "duration_ms": 300000,
  "summary": {
    "execs_per_sec": 1234.5,
    "total_execs": 100000,
    "corpus_found": 42,
    "saved_crashes": 1,
    "saved_hangs": 0,
    "bitmap_coverage": "12.34%",
    "llm_mutator_hits": 100,
    "llm_mutator_misses": 20
  },
  "artifacts": {
    "fuzzer_stats": "repo/path/fuzzer_stats",
    "crashes": "repo/path/crashes.tar.gz",
    "queue": "repo/path/queue.tar.gz",
    "logs": "repo/path/log.txt"
  },
  "error": null
}
```

Такой результат должен попасть в мастер-сервис, а затем стать видимым в UI.

### 20.4. Stateless-подход

Исполняющий fuzzing-сервис не должен хранить:

- описание пайплайнов;
- историю всех запусков как основной источник истины;
- учетные записи пользователей;
- права доступа;
- долгоживущие API keys.

Он может временно хранить:

- рабочий каталог запуска;
- AFL++ output;
- runtime sockets;
- временные generated/feedback samples;
- локальные логи процесса.

После завершения задания постоянные данные должны быть переданы управляющему уровню или во внешнее хранилище артефактов.

## 21. Метрики и отчетность

### 21.1. Уже доступные данные

AFL++ создает `fuzzer_stats`, queue, crashes, hangs и служебные output directories.

Worker сохраняет:

- feedback samples в `runtime/discovered`;
- optionally raw generated candidates в `LLM_MUTATOR_LOG_CANDIDATES_DIR`.

Mutator в памяти считает:

- `requests`;
- `hits`;
- `misses`.

Но эти mutator counters пока не экспортируются наружу.

### 21.2. Метрики из PRD

Минимальные метрики успеха:

- `make all` собирает target и mutator;
- `make smoke` собирает host target и mutator;
- `scripts/run_fake.sh` запускает AFL++ и worker без внешнего API;
- custom mutator работает при доступном и недоступном worker;
- feedback samples появляются в `runtime/discovered`;
- скрытый crash-path достижим;
- `scripts/run_real_llm.sh` работает с OpenAI-compatible endpoint.

Сравнительные метрики:

- time-to-first-crash для baseline AFL++ и AFL++ с LLM-mutator;
- количество unique paths за фиксированное время;
- количество queue entries;
- доля syntactically valid inputs;
- hit/miss ratio очереди LLM-кандидатов;
- overhead custom mutator относительно baseline AFL++.

### 21.3. Метрики, которые еще нужно добавить

Из `TASKS.md`:

- экспорт mutator counters: `requests`, `hits`, `misses`, hit ratio;
- выбор формата экспорта: stderr summary, файл в `runtime/` или AFL-friendly output;
- worker counters: produced, served, queue misses, feedback accepted/rejected, LLM errors;
- исправление пустого `producer error` при заполненной очереди worker;
- compare runner для baseline AFL++ vs LLM-mutator;
- парсер `fuzzer_stats`;
- формат summary report: execs/sec, corpus_found, edges_found, bitmap_cvg, saved_crashes, hit/miss;
- способ считать syntactically valid inputs для DSL corpus.

## 22. Алгоритм фаззинга из презентации

В презентации фаззинг описан как цикл:

1. выбрать случайный вход;
2. мутировать его;
3. выполнить программу;
4. проверить, появилось ли новое покрытие;
5. если новое покрытие есть - добавить input в corpus;
6. если нового покрытия нет - вернуться к выбору следующего входа.

Также отмечено, что мутации полезно выполнять с учетом структуры входа. Например, для XML не просто менять случайные байты, а добавлять или удалять теги. В текущем проекте эта идея перенесена на toy DSL и LLM-generated DSL programs.

## 23. Почему используется AFL++

AFL++ отвечает за то, что уже хорошо решено классическим coverage-guided fuzzing:

- instrumentation;
- запуск target'а;
- измерение покрытия;
- corpus management;
- scheduling;
- crash/hang triage;
- сохранение queue и crashes;
- словари;
- persistent mode.

Проект не пытается заменить AFL++ новым fuzzing engine. LLM используется как дополнительный источник более структурных inputs.

## 24. Почему LLM вынесена в отдельный worker

LLM-вызовы:

- медленные;
- имеют непредсказуемую latency;
- могут зависеть от сети;
- могут падать по timeout;
- могут стоить денег;
- могут возвращать невалидный текст.

Если вызывать LLM внутри `afl_custom_fuzz()`, AFL++ будет ждать модель на каждом шаге и резко потеряет execs/sec. Поэтому выбран асинхронный дизайн:

- worker заранее наполняет очередь;
- mutator быстро получает готовый candidate;
- при miss делает fallback;
- AFL++ не блокируется на HTTP/LLM.

Это главное инженерное решение текущего MVP.

## 25. Безопасность

Текущие правила и свойства:

- API keys не хранятся в репозитории;
- `LLM_API_KEY` передается только через env;
- worker по умолчанию слушает локальный адрес `127.0.0.1`;
- payload sizes ограничены;
- generated `build/`, `output/`, `runtime/` не должны попадать в git;
- vendored AFL++ не меняется без явной задачи;
- реальные endpoints и ключи нельзя документировать в scripts, README или output artifacts.

Для будущего микросервиса дополнительно нужны:

- изоляция запуска target'ов;
- лимиты CPU/RAM/time;
- очистка workspace;
- маскирование secrets в логах;
- контроль сетевого доступа target'а;
- проверка артефактов перед публикацией;
- отдельная политика хранения crash inputs, потому что они могут содержать чувствительные данные.

## 26. Надежность и отказоустойчивость

В текущем fuzzing-ядре:

- отсутствие worker'а не останавливает AFL++;
- пустая очередь worker'а приводит к fallback-мутации;
- ошибки socket приводят к reconnect/drop connection;
- reconnect ограничен по частоте;
- worker сохраняет feedback на диск;
- fake mode позволяет запускать pipeline без сети.

В общей системе надежность усиливается за счет:

- Kafka как асинхронной шины;
- независимого масштабирования исполняющих сервисов;
- хранения статусов в мастер-сервисе/БД;
- возможности повторить задание;
- отделения долгих задач от REST-запросов UI.

## 27. Ограничения текущей реализации

Текущее fuzzing-ядро:

- использует toy DSL target, а не реальный production target;
- не содержит Kafka consumer/producer;
- не содержит REST API;
- не интегрировано с PostgreSQL;
- не интегрировано напрямую с менеджером репозиториев;
- не реализует полноценный parser summary для AFL++ результатов;
- не экспортирует hit/miss counters mutator'а;
- не имеет compare runner baseline vs LLM-mutator;
- не deduplicate'ит feedback samples;
- не ограничивает количество persisted feedback samples;
- не валидирует LLM outputs перед отдачей AFL++ сверх простого обрезания размера;
- не удаляет Markdown/code fences из real LLM output;
- не содержит полноценной CI-проверки `make check`.

Эти ограничения не ломают MVP, но важны для перехода от прототипа к полноценному исполняющему микросервису.

## 30. Возможная структура полноценного fuzzing-микросервиса

Текущий runner можно обернуть примерно так:

```text
fuzzing-service/
  src/main/java/... или другой backend runtime
  kafka/
    FuzzingJobConsumer
    FuzzingResultProducer
  runner/
    WorkspaceManager
    ArtifactResolver
    AFLRunner
    ResultCollector
    ReportBuilder
  llm-aflpp/
    Makefile
    src/mutator/
    src/worker/
    targets/
  Dockerfile
```

Зоны ответственности:

- Kafka layer получает и отправляет сообщения.
- Workspace manager создает временные каталоги.
- Artifact resolver скачивает код/артефакты.
- AFLRunner вызывает `make all`, `run_fake.sh`, `run_real_llm.sh` или более низкоуровневую команду `afl-fuzz`.
- Result collector разбирает AFL++ output.
- Report builder формирует JSON/HTML/Markdown summary для мастер-сервиса.
- Dockerfile фиксирует системные зависимости: компиляторы, AFL++, Python, возможно локальный LLM client.

## 31. Риски

Риски из PRD и анализа материалов:

- LLM может генерировать однотипные inputs;
- LLM может нарушать формат;
- внешний API добавляет latency, стоимость и нестабильность;
- слишком частое использование LLM-кандидатов может снижать execs/sec;
- `AFL_CUSTOM_MUTATOR_ONLY=1` отключает полезные стандартные стадии AFL++;
- toy DSL может не показать преимуществ на реальных форматах;
- локальные HTTP proxies могут мешать OpenAI-compatible localhost endpoints;
- без dedupe feedback corpus может разрастаться;
- без метрик сложно доказать преимущество LLM-mutator;
- без изоляции fuzzing target может быть опасен для CI/CD окружения;
- без продуманного контракта Kafka результат fuzzing-сервиса будет трудно отображать в UI.
