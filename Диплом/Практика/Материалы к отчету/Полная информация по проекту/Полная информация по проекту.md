## 1. Назначение документа

Документ фиксирует итоговое описание разработанной системы для дальнейшего использования в дипломной работе, пояснительной записке, презентации и сопроводительных материалах.

Особенностью разработанной системы является наличие исполняющего сервиса фаззинг-тестирования, использующего гибридный подход: классический coverage-guided fuzzing на базе AFL++ дополняется генерацией структурно осмысленных входных данных с помощью LLM.

В документе описаны:

- реализованный сервис фаззинг-тестирования;
- алгоритм LLM-assisted fuzzing;
- итоговая функциональная готовность;
- результаты выполнения задач.

## 3. Общая характеристика проекта

Разработанная система предназначена для автоматизации процессов сборки, тестирования, доставки и развертывания программного обеспечения. Она реализована по микросервисной архитектуре и состоит из управляющей части, набора исполняющих сервисов и интеграций с внешними системами.

Система позволяет пользователю описывать и запускать CI/CD-пайплайны, состоящие из последовательности этапов и заданий. Управляющий сервис принимает пользовательские действия из UI, формирует задания и передает их исполняющим сервисам через шину данных. Исполняющие сервисы выполняют конкретные виды работ: сборку, фаззинг-тестирование, развертывание и запуск пользовательских скриптов.

Отдельный вклад проекта состоит в реализации сервиса фаззинг-тестирования. Этот сервис дополняет классический CI/CD-процесс проверкой приложения на устойчивость к некорректным и неожиданным входным данным. Для повышения эффективности фаззинга в сервис встроен механизм генерации структурированных входных данных с использованием LLM.

## 4. Актуальность

Современная разработка программного обеспечения требует быстрых, воспроизводимых и безопасных процессов выпуска новых версий. CI/CD-системы позволяют автоматизировать сборку, тестирование и развертывание, уменьшая влияние человеческого фактора и ускоряя цикл поставки программного продукта.

Актуальность разработки обусловлена несколькими факторами:

- потребностью российских организаций в отечественных инструментах автоматизации разработки;
- необходимостью снижения зависимости от зарубежных CI/CD-платформ;
- требованиями безопасности и сертификации в регулируемых отраслях;
- необходимостью полного контроля над процессами сборки, тестирования и доставки;
- важностью раннего выявления дефектов и уязвимостей;
- недостаточной представленностью фаззинг-тестирования как штатного этапа в распространенных CI/CD-решениях.

Фаззинг-тестирование особенно важно для поиска ошибок обработки входных данных, сбоев, аварийных завершений и потенциальных уязвимостей. Однако классические мутации часто разрушают структуру сложного входного формата и не позволяют быстро достигать глубокой логики программы. Поэтому в проекте реализован гибридный подход: AFL++ отвечает за покрытие, исполнение и triage, а LLM помогает создавать более осмысленные входные данные.

## 5. Цель проекта

Цель проекта - разработать готовую микросервисную CI/CD-систему с набором исполняющих сервисов, включая сервис фаззинг-тестирования, способный выполнять автоматизированную проверку программных компонентов и использовать LLM для генерации структурированных тестовых входов.

В рамках проекта реализована система, которая:

- предоставляет пользователю UI для работы с пайплайнами;
- использует управляющий сервис как единую точку входа;
- передает задания исполняющим сервисам через Kafka;
- поддерживает выполнение сборочных, тестовых, deployment и script-задач;
- хранит описание пайплайнов и историю выполнения;
- интегрируется с системами контроля версий, менеджером репозиториев и целевыми серверами;
- содержит готовый сервис фаззинг-тестирования;
- поддерживает LLM-assisted генерацию входных данных для фаззинга;
- формирует результаты выполнения заданий для отображения пользователю.

## 7. Общая архитектура системы

Система построена по принципам микросервисной архитектуры. Компоненты разделены по зонам ответственности и взаимодействуют через явно определенные каналы.

Архитектура включает три группы компонентов:

1. Управляющий уровень.
2. Исполняющие сервисы.
3. Внешние системы.

### 7.1. Управляющий уровень

Управляющий уровень состоит из UI, мастер-сервиса и шины данных.

UI - пользовательская часть системы. Через нее пользователь создает, настраивает и запускает пайплайны, отслеживает статусы выполнения, просматривает логи и результаты заданий.

Мастер-сервис - центральный управляющий компонент. Он принимает запросы от UI, проверяет права доступа, управляет описанием пайплайнов, формирует задания, публикует их в шину данных и сохраняет результаты выполнения.

Шина данных - асинхронный транспорт между управляющим сервисом и исполняющими сервисами. В системе используется Kafka. Такой подход позволяет не блокировать мастер-сервис на длительных задачах и масштабировать отдельные исполняющие сервисы независимо.

### 7.2. Исполняющие сервисы

Исполняющие сервисы выполняют конкретные задачи CI/CD-пайплайна. Они работают как stateless исполнители: получают задание, выполняют его, формируют результат и отправляют его обратно управляющему сервису.

В системе реализованы:

- сборочный сервис;
- сервис фаззинг-тестирования;
- сервис непрерывного развертывания;
- сервис исполнения скриптов.

Сборочный сервис отвечает за сборку проекта и подготовку артефактов.

Сервис фаззинг-тестирования выполняет автоматизированную проверку программных компонентов с помощью AFL++ и LLM-assisted генерации входных данных.

Сервис непрерывного развертывания доставляет подготовленные артефакты на целевые серверы и выполняет deployment-сценарии.

Сервис исполнения скриптов выполняет пользовательские сценарии, необходимые для нестандартных этапов пайплайна.

### 7.3. Внешние системы

Система взаимодействует со следующими внешними компонентами:

- системой контроля версий;
- менеджером репозиториев;
- целевым сервером для развертывания.

Система контроля версий хранит исходный код. Сборочный сервис получает из нее код или взаимодействует с промежуточным хранилищем исходников.

Менеджер репозиториев хранит собранные артефакты, результаты заданий, пакеты и сопутствующие материалы.

Целевой сервер для развертывания используется сервисом непрерывного развертывания для доставки и запуска приложения.

## 8. Взаимодействие компонентов

Типовой сценарий работы системы:

1. Пользователь открывает UI и запускает пайплайн.
2. UI отправляет запрос в мастер-сервис.
3. Мастер-сервис проверяет параметры запуска и права доступа.
4. Мастер-сервис создает внутреннее представление запуска пайплайна.
5. Для каждого этапа формируется одно или несколько заданий.
6. Задания публикуются в Kafka.
7. Исполняющий сервис получает задание из соответствующего топика.
8. Исполняющий сервис выполняет работу и взаимодействует с нужными внешними системами.
9. По завершении исполнитель отправляет статус, логи, метрики и ссылки на артефакты.
10. Мастер-сервис сохраняет результат и обновляет состояние пайплайна.
11. UI отображает пользователю итог выполнения.

Асинхронная модель взаимодействия обеспечивает:

- устойчивость к длительным операциям;
- независимое масштабирование исполнителей;
- отсутствие жесткой связности между сервисами;
- возможность повторной обработки заданий;
- сохранение управляемости при росте нагрузки.

## 9. Модель данных

Для описания пайплайнов и результатов выполнения используется реляционная модель данных.

Основные сущности:

| Сущность | Назначение |
| --- | --- |
| `pipeline` | Описание CI/CD-конвейера. |
| `stage` | Этап внутри пайплайна. |
| `job` | Конкретная работа внутри этапа. |
| `job_history` | История выполнения задания. |
| `job_template` | Шаблон задания определенного типа. |
| `job_params` | Параметры, переданные в задание. |
| `trigger` | Условие или способ запуска пайплайна. |
| `folder` | Логическая группировка пайплайнов. |

`pipeline` хранит название, описание и связь с условиями запуска.

`stage` задает порядок выполнения этапов внутри пайплайна.

`job` описывает конкретную работу: сборку, фаззинг, развертывание или выполнение скрипта.

`job_history` хранит время запуска, продолжительность, логи, статус и дополнительные данные.

`job_template` и `job_params` позволяют переиспользовать типовые задания с разными параметрами.

Такая модель позволяет описывать пайплайны различной сложности, сохранять историю запусков и отображать пользователю детальные результаты выполнения.

## 12. Исполняющий сервис фаззинг-тестирования

Сервис фаззинг-тестирования является завершенным исполняющим микросервисом системы. Он получает задачу от управляющего уровня, выполняет фаззинг заданного target'а, собирает результаты и возвращает их мастер-сервису.

Сервис решает следующие задачи:

- подготовка окружения для fuzzing job;
- запуск target-программы под управлением AFL++;
- подключение custom mutator;
- генерация входных данных в fake или real LLM режиме;
- прием feedback от AFL++;
- сохранение crash cases, queue entries, статистики и логов;
- формирование итогового отчета;
- возврат результата в управляющий сервис.

Внутреннее устройство сервиса разделено на несколько логических частей:

- AFL++ как основной fuzzing engine;
- custom mutator как адаптер между AFL++ и генератором структурированных inputs;
- LLM worker как асинхронный генератор candidates;
- feedback storage для новых интересных входов;
- result collector для формирования итоговой статистики.

## 13. Роль AFL++ в сервисе

AFL++ выполняет базовые функции coverage-guided fuzzing:

- инструментирует target;
- запускает программу на множестве входов;
- измеряет покрытие;
- управляет corpus;
- сохраняет новые интересные inputs;
- выявляет аварийные завершения и зависания;
- выполняет crash triage;
- формирует статистику запуска.

В проекте AFL++ не заменяется собственной реализацией фаззера. Он используется как зрелый и проверенный fuzzing engine. Расширение проекта состоит в том, что AFL++ получает дополнительный источник более структурированных входных данных.

## 14. Custom mutator

Custom mutator реализует связующий слой между AFL++ и генератором входных данных.

Его функции:

- подключение к локальному генератору candidates;
- быстрый запрос готового входа;
- смешивание с текущим seed;
- передача результата AFL++;
- fallback-мутация при недоступности генератора;
- передача новых интересных inputs обратно worker'у;
- сбор hit/miss статистики;
- корректное освобождение ресурсов после завершения.

Mutator работает в hot path AFL++, поэтому его логика построена так, чтобы не выполнять медленные сетевые операции. Он не обращается к LLM напрямую, а получает уже подготовленные candidates от локального worker'а.

## 15. LLM worker

LLM worker - отдельный процесс, который заранее формирует входные данные и держит их в очереди.

Worker поддерживает два режима:

- fake mode;
- real LLM mode.

Fake mode используется для воспроизводимой проверки без внешней модели и API-ключей. В этом режиме worker генерирует синтаксически корректные тестовые программы по локальным правилам.

Real LLM mode используется для генерации входов через OpenAI-compatible endpoint. Worker формирует prompt, добавляет seed examples и feedback examples, отправляет запрос модели и сохраняет результат как candidate.

Worker реализует:

- bounded queue candidates;
- несколько producer-потоков;
- перечитывание prompt при изменении;
- загрузку стартового corpus;
- прием feedback samples;
- дедупликацию feedback;
- ограничение размера и количества persisted samples;
- фильтрацию пустых и нерелевантных samples;
- логирование generated candidates;
- счетчики produced, served, misses, accepted/rejected feedback и LLM errors.

## 16. IPC между mutator и worker

Для связи между AFL++ custom mutator и LLM worker реализован локальный IPC-протокол.

Протокол включает две операции:

- `G` - получить готовый candidate;
- `A` - добавить feedback sample.

Операция `G` используется mutator'ом в момент, когда AFL++ запрашивает очередную мутацию. Если candidate есть в очереди, worker возвращает payload. Если очередь пуста, возвращается miss, и mutator применяет локальную fallback-мутацию.

Операция `A` используется, когда AFL++ нашел новый интересный input. Mutator отправляет этот input worker'у, после чего sample сохраняется и используется в следующих генерациях.

Такой протокол обеспечивает двусторонний feedback loop:

```text
AFL++ -> custom mutator -> worker -> generated candidate -> AFL++
  ^                                                 |
  |                                                 v
  +------------- new interesting inputs <-----------+
```

## 17. Алгоритм LLM-assisted fuzzing

Работа сервиса фаззинг-тестирования выполняется по следующему алгоритму:

1. Сервис получает fuzzing job от управляющего микросервиса.
2. Подготавливается рабочее окружение, target, seed corpus и параметры запуска.
3. Запускается LLM worker.
4. Worker наполняет очередь структурными candidates.
5. Запускается AFL++ с подключенным custom mutator.
6. AFL++ выбирает seed из corpus.
7. Custom mutator запрашивает candidate у worker'а.
8. Если candidate доступен, mutator использует его для формирования нового входа.
9. Если candidate недоступен, применяется локальная grammar-aware fallback-мутация.
10. Target выполняется на новом входе.
11. AFL++ анализирует покрытие и crash behavior.
12. Если найден новый интересный input, он добавляется в corpus.
13. Новый input отправляется worker'у как feedback.
14. Worker использует feedback в следующих генерациях.
15. По завершении бюджета запуска result collector формирует отчет.
16. Сервис отправляет результат мастер-сервису.

## 18. Генерация входных данных

Сервис поддерживает несколько стратегий генерации:

- генерация синтаксически корректных inputs по локальным правилам;
- генерация через LLM по prompt;
- repair существующего входа;
- extend существующего входа;
- combine нескольких seed examples;
- генерация edge-values;
- grammar-aware fallback при отсутствии candidates;
- post-processing LLM output;
- validation и sanitization перед передачей input в AFL++.

Такой набор стратегий позволяет сочетать скорость локальных мутаций и способность LLM создавать более осмысленные структуры.

## 19. Feedback loop

Feedback loop является одной из ключевых частей сервиса.

После каждого обнаружения нового интересного input AFL++ помещает его в corpus. Custom mutator получает уведомление о новом элементе queue и отправляет sample worker'у. Worker:

- проверяет размер;
- отбрасывает пустые и неинформативные samples;
- выполняет dedupe;
- сохраняет sample в feedback corpus;
- учитывает sample при построении prompt;
- ограничивает объем persisted feedback;
- выделяет crash-inducing samples в отдельную категорию.

Это позволяет LLM worker'у постепенно учитывать фактически найденные AFL++ входы и генерировать новые candidates на основе уже полезных примеров.

## 22. Демонстрационный target

Для проверки работы fuzzing-сервиса реализован демонстрационный target с line-based DSL. Он используется для воспроизводимого подтверждения того, что:

- AFL++ успешно запускает target;
- custom mutator подключается и формирует inputs;
- worker генерирует синтаксически валидные candidates;
- feedback от AFL++ возвращается worker'у;
- hidden crash-path достижим;
- результаты фаззинга сохраняются и анализируются.

DSL target содержит регистры, режимы работы, команды изменения состояния, проверки и специальный crash-path. Для достижения crash требуется осмысленная последовательность команд и значений. Это демонстрирует преимущество структурной генерации над полностью случайными байтовыми мутациями.

Демонстрационный target используется как проверочный стенд. При этом архитектура сервиса не привязана к нему и поддерживает адаптацию под реальные targets, форматы, протоколы и API.

## 23. Адаптация под реальные targets

Такая схема позволяет применять сервис не только к toy DSL, но и к более практическим объектам тестирования:

## 25. Интеграция fuzzing-сервиса в CI/CD-пайплайн

В пайплайне fuzzing-сервис используется как отдельный job или stage. Он может выполняться после сборки и до развертывания.

Типовой pipeline:

```text
Получение кода
  -> Сборка
  -> Unit-тесты
  -> Фаззинг-тестирование
  -> Публикация артефактов
  -> Развертывание
```

Если fuzzing stage обнаруживает crash, пайплайн может:

- завершиться ошибкой;
- сохранить crash artifacts;
- уведомить пользователя;
- сформировать отчет;
- заблокировать deployment;
- создать задачу на исправление.

Если fuzzing stage завершается без критических ошибок, пайплайн продолжает выполнение.

## 30. Производительность

Ключевое производительное решение - вынесение LLM из hot path AFL++.

`afl_custom_fuzz()` не выполняет сетевой запрос к модели. Вместо этого:

- отдельный worker заранее генерирует candidates;
- candidates складываются в локальную очередь;
- mutator быстро забирает готовый input;
- при miss выполняется локальная мутация;
- AFL++ сохраняет высокий темп executions.

Такой подход позволяет использовать преимущества LLM, не превращая каждый fuzzing iteration в медленный HTTP-запрос.
